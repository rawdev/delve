# 01. 데모 목표 — 심문 쿼리와 쿼리 역산

이 문서가 **개발 순서를 정한다**. 게임 기능 목록이 아니라, 여기 적힌 쿼리들이
공개 그래프에서 답을 내는지가 프로젝트의 합격선이다.

## 쿼리 역산이란

> 방문자가 던질 질문 → 그 질문에 답하려면 그래프에 무엇이 있어야 하는가 →
> 그 "무엇"이 개발 과정에서 **실제로 일어나도록** 페이즈를 배치한다.

주의: 사건을 *지어내는* 게 아니라 **실제로 일어날 일을 정하는 것**이다. 지어낸
히스토리는 개발자 눈에 즉시 티가 난다.

프레이밍 원칙: 능력을 과시하는 질문("가장 크게 뒤집힌 결정은?", "누가 반대했나?")은
배제한다. **개발자가 몇 달 뒤 실제로 던질 회상 질문**만 쓴다.

## A. 설계 축 — 결정의 진화를 추적

### DQ1. "전투 시스템 설계가 지금까지 어떻게 바뀌었어?"

- **보여주는 것**: 결정 추적 + 인과. 단일 이벤트가 아니라 *결정의 사슬*.
- **답이 되는 사건**: 즉시판정(lockstep) 턴 처리 → 에너지 스케줄러 전환.
  적 3종에 속도 차이(Rat 150 / Goblin 100 / Golem 60)를 넣는 순간 v1이 구조적으로
  깨진다. → [04_turn_system_pivot.md](04_turn_system_pivot.md)
- **그래프에 있어야 할 것**:
  1. `결정` 이벤트 — "턴 처리 v1: 즉시판정 채택" (대안: 에너지 시스템 — *지금은
     적이 1종이고 속도 개념이 없어서 과설계*라는 기각 이유)
  2. `결정` 이벤트 — "턴 처리 v2: 에너지 스케줄러로 전환" (증상: 속도 다른 적을
     표현 불가 / 대안 3개와 기각 이유 / 전환 비용 / 커밋 해시)
  3. 두 이벤트가 엔티티 **"턴 시스템"** 을 공유 → `temporal_flow`로 순서가 드러남
- **핵심**: v1이 *틀린 결정이었다*가 아니라 *그때는 맞았고 조건이 바뀌었다*가
  기록되어야 한다. 이게 실개발의 모양이다.

### DQ2. "던전 생성 로직 밸런스 잡은 히스토리 보여줘"

- **보여주는 것**: `temporal_flow` 시간축 — 수치의 진화.
- **답이 되는 사건**: 던전 파라미터 v1 → 동료 플레이테스트 → "너무 어려움" →
  일부 롤백 → v2. (Phase 3)
- **그래프에 있어야 할 것**: 각 단계가 **별도 이벤트**여야 한다. 한 이벤트에
  "v1에서 v2로 바꿨다"고 뭉쳐 쓰면 시간축이 안 생긴다.
  - `밸런스` 이벤트: 초기 수치 v1 (방 개수/몬스터 밀도/아이템 스폰율)
  - `밸런스` 이벤트: 동료 플레이테스트 결과 (**동료 계정에서 저장**)
  - `결정` 이벤트: 롤백 판단과 이유
  - `밸런스` 이벤트: v2 수치와 근거
  - 전부 엔티티 **"던전 생성"** 공유

## B. 구현 축 — 누가/무엇을 만들었나 (귀속)

### IQ1. "적 AI 로직은 어떻게 구현됐어?"

- **보여주는 것**: **cross-AI 귀속**. 구현 질문 하나에 "누가 설계하고 누가 짜고
  누가 리뷰했나"가 딸려 나온다.
- **답이 되는 사건**: 적 AI 한 기능을 서로 다른 AI가 나눠 맡는다.
  | 단계 | 담당 | 저장 이벤트 |
  |---|---|---|
  | 설계 | ChatGPT | `결정` — 적 AI 상태머신 설계 (idle/chase/flee), 대안 기각 이유 |
  | 구현 | Claude Code | `구현` — 코드 + 커밋 해시 (co-author 컨벤션) |
  | 리뷰 | Gemini | `리뷰` — 지적 사항, 수정 여부 |
- **그래프에 있어야 할 것**: 세 이벤트가 전부 엔티티 **"적 AI"** 를 공유. 저장은
  각 AI의 세션에서 각각 이루어져야 작성자 귀속이 다르게 찍힌다. (한 세션에서
  몰아 저장하면 전부 같은 작성자가 되어 IQ1이 죽는다.)

### IQ2. "@동료가 세이브/로딩 쪽에서 짚거나 고친 게 뭐였어?"

- **보여주는 것**: **cross-user 내용 검색** — `@handle` prefix 검색.
- **답이 되는 사건**: 동료의 **실제 기여** — 코드 리뷰 지적, 플레이테스트 피드백,
  직접 수정 PR. 반대·논쟁을 연출하지 않는다.
- **그래프에 있어야 할 것**: 동료가 **자기 계정에서** 저장한 이벤트. 부계정이라도
  별칭을 설정해 작성자가 분리되어야 한다. → [08_participants_workflow.md](08_participants_workflow.md)
- **주의**: 동료가 없으면 "여러 사람이 함께 쌓는 공유 메모리" 포지셔닝이 그래프에서
  아예 안 보인다. **데모의 존재 이유가 빠진다.** 부계정을 써서라도 확보한다.

## C. 버그 축 — 쿼리는 열어두고, 버그는 실제로 나온 걸 쓴다 (승부처)

> **승부처는 "특정 심은 버그"가 아니라 "저장 규약"이다.**
> 버그를 고칠 때마다 *증상 + 근본 원인(원인이 된 과거 이벤트·결정 링크)* 을 같이
> 저장하는 것 — 이건 그냥 좋은 디버깅 관행이라 작위성이 0이고, **어떤 버그가
> 나오든 그래프가 인과를 물어올 준비가 된다.**
> "그냥 검색이랑 뭐가 달라?"를 끝내는 건 키워드가 **구조적으로 실패하는** 인과·연결
> 쿼리뿐이다.

### BQ1. "가장 최근 버그랑 그 원인이 뭐야?"

- 쿼리가 **특정 버그를 전제하지 않는다.** 그 시점에 실제로 마지막에 고친 버그가
  답이 된다.
- 원인이 최근 커밋이면 그것대로 답이고, 과거 설계 결정까지 거슬러 올라가면
  **인과 도약이 저절로 드러난다** — 심은 게 아니라 발견된 것.
- 성립 조건: [06_memory_protocol.md](06_memory_protocol.md)의 버그 이벤트 4요소를
  **예외 없이** 지킬 것.

### BQ2. "최근에 고친 버그 중에 예전 설계 결정이 원인이었던 거 있어?"

- **인과 도약** — 키워드 검색으로는 구조적으로 불가능한 질문. (버그 텍스트와 원인
  결정 텍스트에 겹치는 단어가 없기 때문.)
- "그 버그"를 지정하지 않는다. 실개발에서 이런 케이스는 자연히 몇 개 생기고,
  그중 하나가 시연 대상이 된다.
- **없으면 없는 대로 정직.** 억지로 만들지 않는다.
- 후보(예상되지만 **보장하지 않음**): 인메모리 dict 상태 → 세이브 불가 / 시드
  미보존 → 리플레이 불일치 / 즉시판정 잔재 → 가속 포션 버그. 이건 예측이지
  계획이 아니다.

### BQ3. "인벤토리랑 세이브 포맷이 왜 같은 시점에 같이 바뀌었어?"

- **보여주는 것**: 공유 엔티티를 통한 그래프 연결. git 히스토리를 읽다 실제로
  나오는 "왜 이 둘이 같이 변경됐지?" 질문.
- **답이 되는 사건**: 인벤토리를 구현하면 직렬화 스키마가 바뀐다. 필연이다.
- **그래프에 있어야 할 것**: 인벤토리 이벤트와 세이브 이벤트가 엔티티
  **"세이브 포맷"** 을 공유. 이 엔티티를 **Phase 2에서 미리 명명**해 두는 것이
  준비의 전부다. (버그를 심는 게 아니라 **이름을 일관되게 쓰는 것** — 규약이지 연출이 아니다.)

## 전 구간 공통

### PQ. "이 결정/수정이 들어간 커밋 보여줘"

- **보여주는 것**: GitHub provenance. 이벤트 → 코드.
- **성립 조건**: 모든 결정·버그수정 이벤트에 커밋 해시를 붙이면 **자동 성립**한다.
  → [07_github_provenance.md](07_github_provenance.md)
