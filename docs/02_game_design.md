# 02. 게임 설계 — Delve

코드명 **Delve**. 웹 브라우저 턴제 로그라이크. **재미가 아니라 완주 가능성이
목표다.** 스코프를 늘리고 싶은 충동이 들면 [00_overview.md](00_overview.md)의
"작지만 진짜" 원칙으로 돌아온다.

## 1. 코어 루프

```
층 진입 → 절차 생성된 던전 탐색(FOV) → 적과 조우 → 턴제 전투
       → 아이템 획득/사용 → 계단 발견 → 하강 → (5층에서 보스) → 클리어 or 사망
사망 = permadeath: 세이브 삭제, 처음부터
```

한 칸 이동 = HTTP 요청 1개 → 서버가 턴을 진행 → JSON 상태 반환 → 브라우저는
그리드를 다시 그린다. 실시간 루프도 웹소켓도 없다. → [03_architecture.md](03_architecture.md)

## 2. 맵과 던전 생성

| 항목 | 값 | 비고 |
|---|---|---|
| 맵 크기 | 64 × 32 타일 | 스크롤 없음 — 화면에 전체가 들어간다 |
| 층 수 | 5 | 5층에 보스 |
| 방 개수 | 6~10 | 층이 깊을수록 +1 경향 |
| 방 크기 | 5~12 (w, h 각각) | |
| 생성 알고리즘 | 랜덤 방 배치 + 겹침 거부 + L자 복도 연결 | BSP는 과설계 — 기각 |
| 타일 종류 | `#` 벽 / `.` 바닥 / `>` 내려가는 계단 | |
| 시야(FOV) | 반경 8, 그림자 캐스팅 | `visible`(현재) / `explored`(가본 곳) 분리 |

**생성은 전적으로 시드 결정론적이다.** 같은 시드 → 같은 던전. 엔티티 이름은
항상 **"전역 시드"** 와 **"던전 생성"**.

### 밸런스 파라미터 v1 (Phase 3에서 이 수치를 흔든다 — DQ2)

```python
# 밸런스 v2 — 초반 Rat 압박 → 후반 Golem 비중. v1에서 1층 Golem을 제거했다
# (플레이테스트 시드 20260716에서 1층 조기 조우가 치명적 → 롤백).
FLOOR_PARAMS = {
    1: {"rooms": (6, 8),  "enemies": {"rat": 2, "goblin": 1}, "items": 3},  # v2: Golem 제거
    2: {"rooms": (6, 9),  "enemies": {"rat": 2, "goblin": 3, "golem": 1}, "items": 3},
    3: {"rooms": (7, 9),  "enemies": {"rat": 2, "goblin": 4, "golem": 2}, "items": 2},
    4: {"rooms": (7, 10), "enemies": {"rat": 2, "goblin": 4, "golem": 4}, "items": 2},
    5: {"rooms": (5, 6),  "enemies": {"rat": 1, "goblin": 2, "golem": 3}, "items": 2},  # 보스는 이후
}
```

> 이 수치는 **틀릴 것이다.** 그게 정상이고, 틀린 걸 동료 플레이테스트로 발견해
> 롤백하고 v2를 내는 과정 자체가 DQ2의 답이다. 미리 완벽하게 맞추려 하지 말 것.

## 3. 액터 (플레이어 + 적)

모든 액터는 같은 데이터 구조를 공유한다. **플레이어도 액터 중 하나다** —
이것이 [04_turn_system_pivot.md](04_turn_system_pivot.md)의 에너지 스케줄러를
가능하게 하는 전제.

| 필드 | 의미 |
|---|---|
| `id`, `kind`, `glyph` | 식별자 / 종류 / 표시 문자 |
| `x`, `y` | 위치 |
| `hp`, `max_hp` | 체력 |
| `atk`, `def_` | 공격력 / 방어력 |
| `speed` | **에너지 회복량** — v2에서 도입 (v1엔 없다) |
| `energy` | **행동 게이지** — v2에서 도입 |
| `ai_state` | `idle` / `chase` / `flee` (적 전용) |

### 적 3종 — 속도 차이가 핵심

| 적 | glyph | HP | ATK | DEF | **speed** | XP | 행동 |
|---|---|---|---|---|---|---|---|
| Rat | `r` | 4 | 2 | 0 | **150** | 3 | 시야 안이면 무조건 추격. 빠르다 |
| Goblin | `g` | 10 | 4 | 1 | **100** | 8 | 표준. HP 30% 이하면 `flee` |
| Golem | `G` | 26 | 8 | 4 | **60** | 20 | 느리지만 아프다. 절대 도망 안 감 |

> **`speed` 열이 이 프로젝트의 유일한 사전 설계 통과 지점이다.**
> Rat이 플레이어보다 1.5배 빠르고 Golem이 0.6배 느리다는 것 — 즉시판정(1입력 =
> 모든 적 1행동) 구조로는 **표현할 방법이 없다.** Phase 2에서 이 3종을 넣는 순간
> 턴 시스템을 다시 짜야 한다. 이건 로그라이크를 만들면 반드시 겪는 일이지
> 연출이 아니다.

### 보스 — Delve Warden (5층)

| glyph | HP | ATK | DEF | speed | 특수 |
|---|---|---|---|---|---|
| `W` | 60 | 10 | 5 | 100 | HP 50% 이하에서 speed 150으로 각성 |

보스의 "각성"은 **에너지 시스템이 있어야만 성립한다.** v1 구조에서는 구현 불가.
전환의 필요성을 두 번째로 증명하는 장치.

### 플레이어

- 시작: HP 20 / ATK 5 / DEF 2 / speed **100** / 시야 8
- 레벨업: XP `10 * level` 누적 시 → HP +5, ATK +1, 2레벨마다 DEF +1
- permadeath: 사망 시 세이브 삭제

## 4. 전투

```
데미지 = max(1, 공격자.atk - 방어자.def_)
```

명중 판정 없음(항상 명중), 크리티컬 없음. **의도적으로 단순하다** — 전투의
복잡도는 수치가 아니라 **턴 순서**에서 나온다. 그게 이 게임의 유일한 전술적 깊이고,
동시에 전환점이 겨냥하는 지점이다.

이동하려는 칸에 적이 있으면 = 공격 (별도 attack 명령 없음, 로그라이크 관례).

## 5. 아이템과 인벤토리

| 아이템 | glyph | 효과 |
|---|---|---|
| 포션 | `!` | HP +10 (최대치 초과 불가) |
| 검 | `/` | 장착 시 ATK +3 |
| 방패 | `[` | 장착 시 DEF +2 |
| 귀환 스크롤 | `?` | 현재 층의 시작 지점으로 순간이동 |

- 인벤토리 슬롯 **10개**. 초과 시 획득 불가.
- 장착 슬롯: 무기 1 / 방어구 1.
- 아이템은 바닥에 놓여 있고 `g`(pickup)로 줍는다.

> **인벤토리 구현 = 세이브 포맷 변경이다.** 상태에 `inventory[]`와 `equipped{}`가
> 추가되면 직렬화 스키마가 바뀐다. 이 둘을 **같은 엔티티("세이브 포맷")로 저장**하는
> 것이 BQ3의 전부다. → [01_demo_goals.md](01_demo_goals.md) BQ3

## 7. 렌더링

1. **1단계 (Phase 1~3)**: 텍스트 그리드. `<pre>` 한 덩어리. `@` 플레이어, `#` 벽,
   `.` 바닥, `r`/`g`/`G` 적.
2. **2단계 (선택)**: 이모지 타일 또는 Canvas.
3. **에셋 0개로 시작한다.** 로그라이크는 그래픽 없이 태어난 장르 — 이 프로젝트가
   로그라이크인 이유 중 하나.

FOV 표현: `visible` = 밝게, `explored`(현재 안 보임) = 어둡게, 미탐색 = 공백.
